# Deployment

Pixie ships as a single podman-compose service. The `pixie-lab`
CLI writes the compose file + envvars into a directory of the
operator's choosing.

## Directory layout

The recommended layout on the deploy host:

```
/opt/pixie/
    compose.yml
    envvars
    envvars.example
    data/                  <- persistent state (state.db, blobs, artifacts)
        live-env/          <- staged pixie live env (vmlinuz + initrd + squashfs)
```

`compose.yml` is generated by `pixie-lab init` and shouldn't need
hand-edits for a standard deploy. `envvars` carries the site-specific
values.

## Lifecycle

The `pixie-lab` CLI drives a deploy directory. Install it with
`uv tool install pixie-lab` (or `pipx install pixie-lab`); each
subcommand takes an optional path, defaulting to the current
directory.

### First deploy

```
pixie-lab deploy /opt/pixie
```

`deploy` writes `compose.yml` + `envvars` (auto-filling a random admin
password and the detected host address), brings the stack up with
`podman compose`, and waits for `/healthz`. It refuses to overwrite an
existing `compose.yml` / `envvars`, so it is a fresh-directory command;
pass `--admin-password` / `--host-addr` to pin those values, or edit
`envvars` and re-run `podman compose --env-file envvars up -d`.

### Restart vs. teardown

A restart is safe for running targets: pixie respawns every nbdkit +
qemu-nbd export from `state.db` on startup, and the nbdboot initrd
connects with `nbd-client -persist`, so a target whose root is on
pixie's NBD reconnects across a quick bounce.

```
podman restart pixie
```

Taking the container fully down drops those exports. Any target
currently booted `nbdboot` off pixie loses its backing device and
hangs - power such targets off before a teardown, or reboot them after.

### Purge and start from scratch

```
pixie-lab purge /opt/pixie --data
```

`purge` stops and removes the container. It prints what it will do and
asks for a `y/N` confirmation first; pass `-y` / `--yes` for scripted
runs (on a non-TTY it refuses without `--yes` rather than fire
unattended). Flags choose how far the wipe goes:

- `--data` - also delete `data/`: `state.db` (catalog, machines,
  events, exports, overlays, settings), every fetched blob and unpacked
  artifact, all per-machine overlays, and the staged `live-env/`.
  `data/` is a bind mount (`./data:/var/lib/pixie`), not a named
  volume, so purge unlinks it directly - `podman compose down -v`
  alone would not touch it.
- `--images` - also remove the pixie container image.
- `--all` - also delete the deploy directory (`compose.yml`,
  `envvars`); implies `--data`.

With no flag, purge only stops the container and leaves state in place.
Bring a config-preserved deploy back up with `podman compose --env-file
envvars up -d`. For a from-nothing redeploy - regenerated compose +
envvars + a fresh admin password - purge with `--all`, then
`pixie-lab deploy /opt/pixie` on the now-empty directory.

### Upgrade the image

The service runs the `pixie:local` image. Rebuild and recreate; state
in `data/` survives untouched:

```
podman build -t pixie:local -f Containerfile .
cd /opt/pixie && podman compose --env-file envvars up -d
```

### Stage the live env

The `pixie-flash-*`, `pixie-inventory`, and `pixie-tui` boot modes chain
into pixie's live env (`vmlinuz` + `initrd` + `live.squashfs`). Click
**Fetch live-env** on the dashboard Live-env card and pixie pulls the
netboot-pc bake from `PIXIE_LIVE_ENV_SRC` (default: the latest pixie
GitHub release's `pixie-live-env-x86_64.tar.gz`) and stages it under
`PIXIE_LIVE_ENV_DIR` itself - no manual bake or copy. For an air-gapped
install, point `PIXIE_LIVE_ENV_SRC` (or the live override on
Settings > Live-env) at a local mirror of that tarball. You can still
build it yourself with `make build VARIANT=netboot-pc` and drop the
three files into `data/live-env/` by hand.

## Environment variables

Pixie reads these at container startup. The compose file passes the
important ones from `envvars`; the rest fall through to defaults.

### Required

**`PIXIE_ADMIN_PASSWORD`** - password for the `/ui/login` form. Pixie
refuses UI writes when unset. The default `envvars` file ships with
a random placeholder; replace it before exposing pixie beyond
`127.0.0.1`.

**`PIXIE_HOST_ADDR`** - LAN hostname or IP pixie tells iPXE targets
to fetch artifacts from. Set to the pixie host's address on the
lab network so target PXE firmware can reach it.

### Common tuning

**`PIXIE_PUBLIC_HOST`** (default: `PIXIE_HOST_ADDR`) - override if
pixie is behind a reverse proxy for HTTP.

**`PIXIE_NBD_PUBLIC_HOST`** (default: `PIXIE_PUBLIC_HOST`) - override
if HTTP is proxied but NBD is direct.

**`PIXIE_LIVE_ENV_EXTRA_CMDLINE`** - kernel-cmdline tokens appended
to every pixie-live-env chain. Common values are hardware-quirk
workarounds; see [](hardware-quirks.md).

**`PIXIE_DATA_DIR`** (default: `/var/lib/pixie`) - state root. Holds
`state.db`, `blobs/` (fetched image bytes), `artifacts/` (extracted
netboot bundles), `overlays/<mac>/<image_sha>/<profile>.qcow2`
(per-machine writable overlays for nbdboot; see [](boot-modes.md)),
`live-env/` (staged pixie live env).

**`PIXIE_LIVE_ENV_DIR`** (default: `$PIXIE_DATA_DIR/live-env`) - the
`pixie-*` boot modes degrade to an unavailable plan when this dir is
missing or incomplete.

**`PIXIE_LIVE_ENV_SRC`** (default: the latest pixie GitHub release's
`pixie-live-env-x86_64.tar.gz`) - the tarball the dashboard **Fetch
live-env** action downloads and stages into `PIXIE_LIVE_ENV_DIR`.
Accepts `https://` or `oras://`; it must unpack to `vmlinuz` + `initrd`
+ `live.squashfs`. Overridable live on Settings > Live-env.

**`PIXIE_SEED_CATALOG`** (default: `1`) - seed a fresh (empty) catalog
from the bundled curated catalog (the netboot-capable nosi subset) on
first start. Set to `0` to start with an empty catalog. Runs once and
never overwrites an operator-populated catalog.

### NBD supervisor

**`PIXIE_NBD_BIND`** (default: `0.0.0.0`) - address nbdkit binds
per-image exports to. Under `--network=host` on compose, `0.0.0.0`
reaches the LAN.

**`PIXIE_NBD_PORT_BASE`** (default: `10809`) - first TCP port
reserved for exports; each subsequent export lands on the next free
port.

### TFTP responder

**`PIXIE_TFTP_ENABLED`** (default: `false`) - set to `1` / `true` /
`yes` / `on` to bring pixie's in-process TFTP responder up on
startup. Requires root; the compose file's `--network=host` running
as root inside the container handles this.

**`PIXIE_TFTP_BIND`** (default: `0.0.0.0`)
**`PIXIE_TFTP_PORT`** (default: `69`) - PXE firmware assumes udp/69.

The Settings page in the pixie UI (`/ui/settings#deployment`)
documents all env vars with their live-resolved values on the running
deploy.

## DHCP handoff

Pixie chains iPXE via TFTP: the DHCP server tells the target's PXE
firmware where to fetch the iPXE bootloader, and iPXE then chains
to pixie's HTTP endpoint. The user-class re-DHCP dance lets one
DHCP config serve both PXE ROM firmware (first pass) and iPXE
firmware (second pass).

### Legacy BIOS / PXE

```
# dnsmasq
dhcp-boot=tag:!ipxe,undionly.kpxe,${PIXIE_HOST_ADDR},${PIXIE_HOST_ADDR}
dhcp-userclass=set:ipxe,iPXE
dhcp-boot=tag:ipxe,http://${PIXIE_HOST_ADDR}:8080/pxe-bootstrap.ipxe
```

### UEFI

```
# dnsmasq
dhcp-boot=tag:!ipxe,ipxe.efi,${PIXIE_HOST_ADDR},${PIXIE_HOST_ADDR}
dhcp-userclass=set:ipxe,iPXE
dhcp-boot=tag:ipxe,http://${PIXIE_HOST_ADDR}:8080/pxe-bootstrap.ipxe
```

Same shape for both; the difference is the bootfile name in the
first DHCP pass (`undionly.kpxe` for BIOS, `ipxe.efi` for UEFI).
Pixie serves both over its TFTP responder from a package-bundled
directory.

## Bootstrap URL

Everything after the iPXE handoff is HTTP against
`http://<PIXIE_HOST_ADDR>:8080/pxe-bootstrap.ipxe`, which chains to
the per-MAC plan at `/pxe/<mac>`. No TFTP dependency past the
initial iPXE fetch.
